    **Step 2: Investigate and fix the root cause.**

    The most likely root cause of the 65.9% area loss on panel 8 of the hip-and-valley roof: panel 8 shares edges with 3+ neighbors (multi-neighbor panel at a valley intersection). The current algorithm processes each graph_edge independently and inserts vertices into the target polygon. When the SAME target polygon edge segment receives insertions from multiple source polygons across different graph_edge iterations, the polygon is rebuilt with `np.array(new_verts)` each time -- but the SECOND iteration operates on the ALREADY-MODIFIED polygon from the first iteration, potentially inserting vertices at wrong positions because the edge indices have shifted.

    The fix: after inserting vertices from one graph_edge, the polygon vertex array grows and edge indices shift. Subsequent graph_edge iterations for the same target panel now walk stale indices. The critical fix is to ensure that vertex insertions from multiple source polygons are accumulated correctly.

    Specifically, examine the inner loop structure:
    ```python
    for target_pid, source_pid in [(pid_a, pid_b), (pid_b, pid_a)]:
        target_poly = out[target_pid]  # THIS RE-READS from out[], so it gets the latest
        ...
        out[target_pid] = np.array(new_verts)  # THIS OVERWRITES
    ```

    The `out[target_pid]` IS re-read each iteration, so the second graph_edge DOES see the modified polygon. But the problem is that the inserted vertices from a PREVIOUS graph_edge may now project onto an edge segment that they should NOT be inserted into again (a previously-inserted vertex from panel B is now in panel A's polygon, and when processing the A-C edge, that B-vertex gets re-inserted or causes geometry corruption).

    **The fix approach:** Add a guard in the insertion loop to skip source vertices that are ALREADY present in the target polygon (within tolerance). Before checking if source vertex `q` should be inserted on target edge `p0->p1`, check if `q` (in XY) is already within `tol` of ANY existing vertex in the target polygon. If so, skip it.

    Add this check inside the `for q in source_poly:` loop, BEFORE the `_point_to_segment_dist_xy` call:

    ```python
    # Skip source vertices already present in the target polygon
    # (may have been inserted by a prior graph_edge iteration)
    dists_to_existing = np.linalg.norm(target_poly[:, :2] - q[:2], axis=1)
    if float(np.min(dists_to_existing)) <= tol:
        continue
    ```

    WAIT -- this would also skip legitimate shared-edge vertices. The existing endpoint dedup already handles p0 and p1. The issue is more subtle.

    **Alternative root cause analysis:** The real problem may be that densify inserts a vertex that projects onto the edge in XY but creates a self-intersection in the polygon's actual geometry. When a panel has edges with multiple neighbors, vertex insertions from one neighbor can push the polygon boundary across another edge, creating a self-intersection that Shapely's make_valid() resolves by splitting into a MultiPolygon, discarding 65.9% of the area.

    **Concrete fix (narrow, per D-06):** After all insertions for a given target_pid within a single graph_edge, verify that the new polygon does NOT self-intersect in 2D. If a specific insertion causes self-intersection, skip that insertion.

    **Most likely concrete fix:** The issue is that `_point_to_segment_dist_xy` uses `tol` (1.0m) as the maximum distance from source vertex to target edge for insertion eligibility. But on a multi-neighbor panel at a valley, source vertices from a non-adjacent neighbor may be within 1.0m of a target edge they don't actually share. The fix: only insert source vertices that lie near the SHARED EDGE between the two specific panels (defined by the feature endpoints), not near ANY edge of the target polygon.

    **Implementation of the fix:**

    Before the inner `for i in range(n):` loop that walks target polygon edges, identify which target polygon edges are actually part of the shared edge between pid_a and pid_b. Use the graph_edge's `feature_ids` to find the feature positions (the endpoints of the shared edge). Then ONLY insert source vertices onto target edges whose endpoints are near these feature positions.

    Add this logic:

    ```python
    # Identify the shared edge endpoints from the feature graph
    shared_features = [
        f for f in graph.get("features", [])
        if f["id"] in graph_edge["feature_ids"]
    ]
    shared_positions = []
    for f in shared_features:
        if f.get("position_xyz"):
            shared_positions.append(np.array(f["position_xyz"][:2]))

    # For each target polygon edge, only consider insertions if BOTH
    # endpoints of the target edge are near a shared feature position
    # (within 2*tol to account for solved positions not being exact)
    ```

    However, this approach depends on position_xyz being filled in, which happens AFTER solving. Since densify runs after solve_apices, positions should be available via the solved polygon vertices but NOT necessarily in the graph features (position_xyz may still be None until _update_graph_positions runs after densify).

    **Simplest correct fix:** Restrict insertions to target polygon edges where AT LEAST ONE endpoint is within `tol` of a vertex in the source polygon. This ensures we only densify edges that are genuinely shared between the two panels.

    In the `for i in range(n):` loop, add a pre-check:

    ```python
    # Only consider this target edge if it's part of the shared
    # boundary: at least one endpoint must be near a source vertex.
    p0_near_source = float(np.min(
        np.linalg.norm(source_poly[:, :2] - p0[:2], axis=1)
    )) <= tol
    p1_near_source = float(np.min(
        np.linalg.norm(source_poly[:, :2] - p1[:2], axis=1)
    )) <= tol
    if not (p0_near_source and p1_near_source):
        continue
    ```

    This requires BOTH endpoints of the target edge to be near source vertices, meaning the entire edge segment is part of the shared boundary. This prevents inserting source vertices onto non-shared edges of multi-neighbor panels.

    **Apply this fix in densify.py.** The check goes at the top of the `for i in range(n):` loop, after computing `p0` and `p1`, before `edge_xy = (p1 - p0)[:2]`.

    D-08: No fallback path. If densify still produces invalid geometry after this fix, the validate pass 2 hard-fails as designed.
